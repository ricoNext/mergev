# Mergev 产品规划

## 一句话介绍

Mergev 是一个在命令行里提供三栏可视化流程的 Git 冲突解决工具，帮助开发者审阅、选择、编辑、校验并完成冲突合并，而无需离开终端。

目标体验接近 WebStorm：逐块选择、编辑并校验最终合并结果。

## 产品定位

Mergev 专注于一个痛点流程：解决 Git 冲突。它把 WebStorm 合并对话框的核心体验带到终端 UI：

- 三栏可视化合并视图
- 按冲突块导航
- 接受 ours、theirs、both，或手动编辑结果
- 实时结果预览
- 将文件标记为已解决前先做校验
- 感知 merge、rebase、cherry-pick 等 Git 流程状态

目标不是替代完整 IDE，而是让主要在终端工作的开发者，能更快、更直观、更可靠地解决冲突。

## 目标用户

- 偏好终端优先 Git 工作流的开发者
- 喜欢 WebStorm 或 VS Code 合并编辑器，但不想离开命令行的开发者
- 在 rebase、cherry-pick、依赖更新中频繁处理冲突的维护者
- 希望冲突解决流程可重复，并带校验命令的团队
- 需要在继续工作前获得结构化、可检查合并结果的 AI coding agent 用户

## 核心问题

Git 内置冲突格式很强大，但不友好：

```text
<<<<<<< HEAD
ours
=======
theirs
>>>>>>> branch
```

开发者需要手动回答这些问题：

- 哪边是 ours，哪边是 theirs？
- 最终文件会长什么样？
- 还有哪些冲突未解决？
- Git 是否自动合并了附近代码——文本上合法，但语义上有风险？
- 结果在语法上是否有效？
- 下一步该跑什么命令：commit、`rebase --continue`，还是 `cherry-pick --continue`？

Mergev 把这些问题变成可交互的工作流。

## 设计灵感

Mergev 受 WebStorm 冲突解决流程启发：

- 开发者从冲突列表打开冲突文件
- 工具展示当前分支、传入分支与最终结果
- 开发者一次解决一个冲突块
- 最终结果始终可见且可编辑
- 工具校验结果，并协助完成当前 Git 操作

## MVP 范围

第一版应完成一条窄但真实的工作流：

> 检测冲突文件，打开三栏终端合并视图，让用户通过接受 ours/theirs/both 或手动编辑逐块解决冲突，写入最终结果，校验，并将文件标记为已解决。

### 1. Git 仓库检测

Mergev 应：

- 检测当前目录是否位于 Git 仓库内
- 找到仓库根目录
- 读取当前 Git 操作状态
- 不在 Git 仓库内时给出清晰错误

相关 Git 状态：

- 普通 merge 冲突
- rebase 冲突
- cherry-pick 冲突
- revert 冲突

### 2. 冲突文件发现

Mergev 应从 Git 列出未合并文件。

对每个文件展示：

- 路径
- 冲突状态
- 检测到的冲突块数量
- 文件类型或扩展名
- 当前会话中是否已解决

示例：

```text
冲突文件

> src/user.ts          3 个冲突   TypeScript
  package.json         1 个冲突   JSON
  pnpm-lock.yaml       1 个冲突   lockfile
```

### 3. 读取 Git 三方版本

Mergev 应从 Git index 读取三个暂存版本：

```bash
git show :1:path   # base
git show :2:path   # ours
git show :3:path   # theirs
```

这比只解析工作区中的冲突标记更可靠。

Mergev 应维护：

- Base 内容
- Ours 内容
- Theirs 内容
- 当前结果内容
- 冲突块元数据
- 每个冲突块的用户决策

### 4. 三栏终端 UI

主文件视图应支持接近 WebStorm 的**外层大模块 + 上中下小模块**布局：

```text
+====================================================================+
| +----------------------------------------------------------------+ |
| | Merge Revisions for src/user.ts                                | |
| | merge  conflict 2/7  unresolved 5                              | |
| +----------------------------------------------------------------+ |
| +--------------+ +--------------+ +--------------+                 |
| | Ours (Left)  | | Result       | | Theirs       |                 |
| | ...          | | ...          | | ...          |                 |
| +--------------+ +--------------+ +--------------+                 |
| +----------------------------------------------------------------+ |
| | [H] Accept Left  [L] Accept Right     [q] Cancel  [s] Apply    | |
| | h/l/b 块级  n/p 导航  e 编辑  ? 帮助                           | |
| +----------------------------------------------------------------+ |
+====================================================================+
```

- 外层：整个合并视图作为一个大模块（圆角边框）
- 上：merge 标题与冲突进度（小模块）
- 中：三栏操作区（撑满顶/底之间剩余空间；无外框，仅各栏有边框；内容超出时鼠标滚轮滚动）
- 下：Accept Left / Accept Right / Cancel / Apply 操作栏（小模块）

中间区滚动：

- 鼠标滚轮：纵向滚动
- Shift+滚轮 / 水平滚轮：横向平移
- 切换冲突（`n` / `p` / `g`）时重置滚动位置
- 不使用方向键/`j``k` 滚动（列表页的 `j`/`k` 仍用于移动选中项）

布局应随终端宽度自适应：

- 宽终端：三栏模式
- 中等终端：双栏模式
- 窄终端：以结果为主的单栏模式

### 5. 冲突导航

用户应能在冲突间移动，而无需手动搜索。

必需操作：

- 下一个冲突
- 上一个冲突
- 跳到第一个未解决冲突
- 在文件间跳转
- 显示当前冲突序号
- 显示未解决数量

建议快捷键：

```text
n    下一个冲突
p    上一个冲突
f    文件列表
g    第一个未解决冲突
?    帮助
```

### 6. 接受变更

对每个冲突块，用户应能：

- 接受 ours
- 接受 theirs
- 接受 both
- 手动编辑
- 将块重置为未解决
- 撤销上一次决策

整文件级操作（底栏）：

- Accept Left：全部冲突接受 ours
- Accept Right：全部冲突接受 theirs
- Cancel：放弃当前文件合并视图（有未保存决策则确认）
- Apply：保存并标记已解决（等同保存流程）

建议快捷键：

```text
h    接受当前块 ours
l    接受当前块 theirs
b    接受当前块 both
H    Accept Left（全部）
L    Accept Right（全部）
e    编辑结果
u    撤销
r    重置当前冲突
q    Cancel
s    Apply
```

每次决策后，结果栏应立即更新。

### 7. 结果预览

结果栏是最终写入工作区文件的唯一真相来源。

它应展示：

- 已解决的冲突内容
- 未解决冲突占位
- 手动编辑内容
- 高亮变更行
- 可用时的校验诊断信息

用户应始终清楚最终会写入磁盘的内容。

### 8. 保存并标记为已解决

当文件没有未解决冲突块时，Mergev 应允许保存。

保存流程：

1. 将结果内容写入工作区文件
2. 确认冲突标记已消失
3. 运行文件级校验
4. 可选运行配置的命令
5. 除非选项禁用，否则执行 `git add path`
6. 更新文件列表状态

建议命令：

```bash
mergev
mergev src/user.ts
mergev --no-add
```

### 9. 基础校验

MVP 校验应包括：

- 无残留冲突标记
- `.json` 的 JSON 解析校验
- `.yaml` / `.yml` 的 YAML 解析校验
- 在支持时做基础 JavaScript / TypeScript 解析校验

校验输出应简短、可行动：

```text
校验: 失败
src/user.ts:42: Unexpected token
```

### 10. Git 流程收尾

全部冲突解决后，Mergev 应检测当前 Git 流程，并建议下一步命令：

- Merge：提交 merge
- Rebase：`git rebase --continue`
- Cherry-pick：`git cherry-pick --continue`
- Revert：`git revert --continue`

后续版本中，Mergev 可提供需确认的动作来继续该流程。

## 超越 MVP 的特性

这些特性让 Mergev 在真实项目中更完整、更好用。

### 1. 非冲突变更审阅

Git 可能成功自动合并部分变更，这些变更在语义上仍可能有风险。

Mergev 应展示附近的非冲突变更，并允许用户在保存前审阅。

可能操作：

- 显示/隐藏非冲突变更
- 应用全部非冲突变更
- 高亮冲突周围的变更上下文

### 2. 行内 Diff

Mergev 应支持行级与词级 diff 高亮。

典型场景：

- 变量重命名
- 函数参数变更
- 字符串变更
- 条件表达式变更
- import 路径变更

### 3. 手动编辑

当无法通过选一边解决冲突时，应支持手动编辑。

可能方案：

- 在 TUI 内编辑当前结果块
- 用 `$EDITOR` 打开当前块或整个结果文件
- 编辑器退出后刷新 TUI
- 编辑后重新运行校验

MVP 可先从外部编辑器支持开始。

### 4. 自定义检查命令

用户应能运行项目特定检查：

```bash
mergev --check "pnpm typecheck"
mergev --check "pnpm test"
mergev --check "npm run lint"
```

配置文件支持可稍后加入。

### 5. 配置文件

Mergev 最终应支持配置文件：

```json
{
  "check": "pnpm typecheck",
  "autoAdd": true,
  "lockfileStrategy": "regenerate",
  "keymap": "default"
}
```

可能的配置位置：

- `.mergev.json`
- `.mergevrc`
- `package.json` 字段：`mergev`

### 6. 会话恢复

大型 rebase 可能耗时较长，用户可能中途退出再继续。

Mergev 应持久化：

- 当前文件
- 当前冲突序号
- 当前会话中的决策
- 未保存的结果内容

会话状态应存放在被 Git 忽略的位置。

### 7. 安全撤销与备份

写入文件前，Mergev 应保护用户工作成果。

可能的安全特性：

- 内存撤销栈
- 保存前备份
- 恢复先前结果
- 退出前展示已变更文件

## 差异化特性

这些特性可让 Mergev 不止是 IDE 合并编辑器的终端翻版。

### 1. 按文件类型感知的冲突处理

Mergev 可为常见文件类型提供更智能的视图。

#### JSON

- 展示对象路径
- 检测重复键
- 校验解析结果
- 按配置排序或保留键顺序

#### JavaScript / TypeScript

- 按函数、类、import 块或 export 分组冲突
- 检测重复 import
- 保存前解析结果
- 在结果栏附近展示语法诊断

#### Markdown

- 按段落或标题区块解决
- 保留周围正文

#### CSS / SCSS

- 按选择器分组
- 检测重复声明

#### Lockfile

- 警告手动解决冲突可能有风险
- 在适当时建议重新生成 lockfile
- 在用户确认后允许文件级接受 ours/theirs

### 2. 语义建议

在引入任何 AI 能力之前，Mergev 可提供基于规则的建议：

- 两边都新增 import：合并并去重
- 两边都新增数组项：在顺序安全时合并
- JSON 对象变更：键不冲突时按键合并
- package scripts 冲突：要求人工审阅

所有建议必须可确认。Mergev 绝不应静默做出有风险的语义决策。

### 3. 可选 AI 辅助

AI 支持应为可选，且属于后期阶段。

可能能力：

- 解释冲突
- 建议合并结果
- 识别可能的语义风险
- 生成已解决文件的摘要

重要约束：

- AI 输出仅是建议
- 必须用户确认
- 结果仍须通过校验
- 本地隐私与 opt-in 行为必须清晰

## CLI 设计

### 基础命令

```bash
mergev
mergev src/user.ts
mergev --check "pnpm test"
mergev --no-add
mergev --list
```

### 未来命令

```bash
mergev continue
mergev abort
mergev status
mergev config init
mergev doctor
```

### 建议选项

```text
--check <command>    完成前运行命令
--no-add             保存后不执行 git add
--all                遍历全部冲突文件
--mode <mode>        强制布局：three-pane、two-pane、result
--editor <command>   手动编辑所用编辑器
--no-session         禁用会话持久化
--debug              显示调试信息
```

## 快捷键模型

初始快捷键：

```text
n    下一个冲突
p    上一个冲突
h    接受 ours
l    接受 theirs
b    接受 both
e    编辑结果
u    撤销

s    保存文件
```

快捷键应在状态栏与帮助屏中可见。

## UI 状态

Mergev 应定义清晰状态：

- 无仓库
- 仓库内无冲突
- 冲突文件列表
- 文件合并视图
- 手动编辑模式
- 校验运行中
- 校验失败
- 文件已保存
- 文件已标记为已解决
- 全部冲突已解决
- Git 流程可继续

## 架构方向

推荐技术栈：

- Node.js
- TypeScript
- Ink（终端 UI）
- execa 或 child_process（Git 命令）
- diff 库（行级与词级 diff）
- 解析库（可选文件校验）

高层模块：

```text
src/
  cli/
  git/
  merge/
  ui/
  validation/
  config/
  session/
```

### Git 层

职责：

- 检测仓库根目录
- 检测冲突状态
- 列出未合并文件
- 读取 stage 1/2/3 内容
- 写入结果文件
- 执行 git add
- 检测 rebase / cherry-pick / merge 状态

### 合并模型层

职责：

- 从 base/ours/theirs 构建冲突模型
- 跟踪决策
- 生成结果内容
- 跟踪未解决冲突
- 支持撤销/重置

### UI 层

职责：

- 渲染文件列表
- 渲染合并视图
- 处理键盘输入
- 显示状态栏
- 显示诊断信息
- 打开帮助屏

### 校验层

职责：

- 检测冲突标记
- 运行文件类型校验器
- 运行自定义命令
- 返回简洁诊断信息

### 会话层

职责：

- 持久化会话状态
- 恢复先前会话
- Git 状态变化时清除会话

## 非目标

第一版不应试图：

- 替代完整文本编辑器
- 替代 WebStorm、VS Code 或 Vim
- 自动解决所有冲突
- 默认提供 AI 合并
- 处理所有二进制或生成文件
- 除非配置，否则不整文件重排格式
- 对外发布或同步任何代码

## MVP 成功标准

若开发者能完成以下步骤，则 MVP 成功：

1. 在有冲突的 Git 仓库中运行 `mergev`
2. 看到冲突文件列表
3. 在三栏终端 UI 中打开文件
4. 按冲突块移动
5. 接受 ours、theirs 或 both
6. 需要时手动编辑
7. 预览最终结果
8. 保存文件
9. 校验冲突标记已消失
10. 用 Git 将文件标记为已解决
11. 理解下一步该运行的 Git 命令

## 开放问题

- 成功保存后，`mergev` 是否默认执行 `git add`？
- `mergev continue` 应执行 Git 继续命令，还是仅建议命令？
- 手动编辑应先在 Ink 内完成，还是 MVP 依赖 `$EDITOR`？
- 默认应展示多少非冲突上下文？
- MVP 是否应将 lockfile 视为特殊文件？
- 配置应先放在 `.mergev.json` 还是 `package.json`？
- 会话状态应存在 `.git/mergev/` 还是项目根目录？
